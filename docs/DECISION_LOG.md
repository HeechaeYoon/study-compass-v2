# Decision Log

이 문서는 원본 Figma와 디자인 자산이 없는 상태에서 Codex가 질문 없이 일관된 결정을 내리도록 하는 고정 판단 기준이다.

---

## D-001. 기준 화면

- 결정: 1280×800을 canonical viewport로 사용한다.
- 근거: 참조 보드 각 패널의 비율이 약 16:10이며, 태블릿 가로 화면과 일반 노트북에 적합하다.
- 보조 검증:
  - 1024×768
  - 1366×768
  - 1440×900
  - 1920×1080
- 1920px에서 콘텐츠를 무한 확장하지 않고 중앙 정렬한다.

---

## D-002. 모바일 범위

- 결정: 900px 미만은 제품 지원 범위에서 제외한다.
- 동작: 화면을 억지로 축소하지 않고 `가로 화면이나 태블릿·PC에서 이용해주세요` 안내를 보여준다.
- 이유: 참조 디자인의 핵심은 넓은 카드 배치와 수평적 정보 구조다.

---

## D-003. 참조 보드 해석

- `reference/design-board.png`는 4개 화면을 한 장에 배치한 디자인 보드다.
- 파란색 `1 시작 화면`, `2 문항 화면` 같은 라벨은 제작용 주석이며 production UI에 넣지 않는다.
- 상단 제목과 메타 정보는 앱 공통 헤더로 구현한다.
- 각 화면은 공통 헤더 아래 하나의 큰 콘텐츠 surface로 구현한다.

---

## D-004. 시각적 동일성의 정의

- 픽셀 100% 동일을 약속하지 않는다.
- 기준 viewport에서 다음이 가까우면 성공으로 판단한다.
  - 전체 구도
  - 카드 비율
  - 여백 밀도
  - 컬러
  - 타이포그래피 계층
  - 그림자와 선 두께
  - 종이·노트·손그림 분위기
- 자동 이미지 비교와 별도 시각 리뷰를 함께 사용한다.

---

## D-005. 문항 선택지 수

- 제품 원칙은 동의형 4점 척도와 상황형 5개 선택지다.
- 참조 문항 화면은 5개 카드의 시각 구성으로 해석한다.
- 실제 컴포넌트는 4개와 5개 선택지를 모두 지원한다.
- 5개 카드가 보이는 Q06 상황형 문항을 배치해 참조와 유사한 실제 화면을 제공한다.
- 동의형에는 중립 선택지를 두지 않는다.

---

## D-006. 기존 구현 처리

- 기존 코드는 참고 자료일 뿐 기본 구조로 간주하지 않는다.
- 기존 CSS 위에 덧대는 방식 금지.
- 질문 데이터나 점수 로직도 현재 문서와 일치하는지 확인 후 재사용한다.
- uncommitted 변경이 있으면 안전하게 보존한 후 교체한다.

---

## D-007. UI 기술

- React + Vite + TypeScript
- plain CSS 또는 CSS Modules
- custom SVG radar chart
- 작은 outline icon은 Lucide 사용 가능
- 전체 UI 라이브러리 금지
- Tailwind는 기존 repo에서 이미 표준일 때만 허용하며, 기본 컴포넌트 외형을 그대로 사용하지 않는다.

---

## D-008. 이미지 생성 사용 범위

- `$imagegen`은 반드시 사용하되, 전체 화면을 이미지로 만들지 않는다.
- 생성 대상:
  - 미세한 종이 질감
  - 오렌지색 나무 연필 장식
  - 필요 시 얇은 종이 가장자리 질감
- 생성하지 않을 대상:
  - 버튼
  - 카드
  - 레이더 차트
  - UI 텍스트
  - 아이콘
  - 전체 노트 패널
- 별, 밑줄, 화살표, 연결선은 SVG로 직접 구현한다.

---

## D-009. 글꼴

원본 글꼴은 확인할 수 없다.

- 본문: Pretendard 계열 우선
- 제목·강조: Gowun Dodum 계열 우선
- 런타임 Google Fonts 요청은 금지
- npm font package 또는 프로젝트 로컬 asset으로 번들
- 실제 캡처에서 참조보다 너무 둥글거나 유치하면 display font를 Pretendard로 되돌리고 손그림 밑줄로 따뜻함을 유지한다.
- 최종 선택은 시각 비교 결과로 결정하고 `VISUAL_REVIEW_LOG.md`에 기록한다.

---

## D-010. 색상과 질감

- 순백보다 따뜻한 ivory 배경
- 핵심 색상은 deep navy와 indigo blue
- 보조 색상은 mint green, warm yellow, coral orange
- 그라데이션은 주요 CTA와 radar fill 정도로 제한
- 종이 질감은 낮은 대비와 낮은 opacity로 사용
- 빈티지, 스크랩북, 과도한 수채화 느낌은 금지

---

## D-011. 차트

- 레이더 차트는 SVG로 직접 그린다.
- 5개 축을 고정 순서로 사용:
  1. 계획
  2. 실행
  3. 이해
  4. 점검
  5. 도움
- 0~100 내부 점수를 차트 반지름에 매핑한다.
- 학생 화면의 부가 숫자는 1~5 범위로 변환해 작은 보조 정보로만 표시할 수 있다.
- 막대나 레이더의 목적은 비교·서열이 아니라 균형 파악이다.

---

## D-012. 결과 숫자 표현

- 내부 계산은 0~100.
- 결과 주 화면에서는 `강점`, `균형`, `성장 포인트`가 중심이다.
- 참조 시안처럼 작은 `4.2` 형태를 사용할 경우:
  - `displayScore = 1 + normalized / 25`
  - 소수점 한 자리
  - 색상과 라벨을 함께 표시
- 숫자만 단독으로 보여주지 않는다.

---

## D-013. 저장

- v1은 저장 결과 최신 1개만 지원해 복잡도를 줄인다.
- 저장은 버튼을 눌렀을 때만 실행한다.
- 시작 화면에서 다시 보기와 전체 삭제 제공.
- localStorage 사용 불가 환경에서도 진단·결과·복사는 동작해야 한다.

---

## D-014. 화면 전환

- React Router는 필요하지 않다.
- 단일 앱 상태 머신으로 화면을 전환한다.
- GitHub Pages 새로고침 문제를 피한다.
- 개발·시각 테스트용 query fixture는 허용한다.

---

## D-015. 애니메이션

- 160~220ms 정도의 미세한 transition만 사용한다.
- 페이지 진입의 큰 motion이나 게임식 효과 금지.
- `prefers-reduced-motion` 지원.
- Playwright screenshot에서는 모든 animation 비활성화.

---

## D-016. 사용자 질문 정책

Codex는 다음 상황이 아니면 사용자에게 질문하지 않는다.

- 참조 이미지 파일을 실제로 읽을 수 없음
- 파일 시스템 쓰기 권한 없음
- 필요한 실행 환경 자체가 없음
- 사용자의 uncommitted 작업을 안전하게 보존할 방법이 없음

색상, 여백, 폰트, 자산 후보 같은 일반적인 디자인 선택은 문서와 visual QA를 통해 스스로 결정한다.

---

## D-017. 완료 선언

다음 중 하나라도 만족하지 않으면 완료를 선언하지 않는다.

- 핵심 4화면 시각 리뷰 90점 미만
- 자동 테스트 실패
- build 실패
- 외부 데이터 전송 존재
- 복사 fallback 부재
- 저장·삭제 미검증
- subagent review 미실행
- review findings 미수정
